Unresolved Symbol cov_probe

SYMPTOMS

The linker reports unresolved symbols with prefix cov_, such as: 

cov_probe_x_y_z

CAUSE

The BullseyeCoverage run-time library is not linked, or the BullseyeCoverage run-time library version number does not match the covc version number.

RESOLUTION

Use one or more of the possible resolutions shown below.

  1. Modify the PATH environment variable so that the BullseyeCoverage/bin directory occurs in the PATH before the real compiler or linker. The simplest way to accomplish this is to put BullseyeCoverage/bin first.

    For example, on Windows: 

    PATH=%ProgramFiles%\BullseyeCoverage\bin;%ProgramFiles%\Microsoft Visual Studio\VC\bin;...

    For example, on Unix: 

    PATH=/opt/BullseyeCoverage/bin:/usr/bin/:...

  2. Modify your makefile or build script to invoke your compiler rather than the linker to perform linking. This resolution does not apply to Microsoft C++, Intel C++, and CodeWarrior. For example, in a makefile: 
    ld ...    # wrong
$(CC) ... # right
  3. Modify your makefile or build script to use the basename of the compiler or linker rather than its absolute path. For example, in a makefile: 
    CC=/usr/bin/cc       # wrong
CC=cc                # right
LINK=c:\bin\link.exe # wrong
LINK=link            # right
  4. Do not use covc option --no-lib.
  5. Remove all BullseyeCoverage run-time libraries then reinstall BullseyeCoverage. This will solve the problem if it is due to a version mismatch.
  6. Explicitly add the BullseyeCoverage run-time library to your link command. This is a last resort unless you have a specially built version of the BullseyeCoverage run-time library. The BullseyeCoverage compiler and linker interceptors automatically modify the link command to add the correct run-time library for the target platform, add required system libraries, and ensure proper ordering of libraries. For Windows DLLs in particular, BullseyeCoverage makes numerous non-trivial changes.
